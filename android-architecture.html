Why Android Architecture Matters

Android is a carefully designed software stack where every layer has a specific responsibility. Instead of allowing apps to communicate directly with hardware, Android routes every request through multiple controlled layers.

What is Android Architecture?

Android Architecture is the layered design that defines how all components of the operating system work together. Each layer communicates only with adjacent layers through well-defined interfaces.

⚠️ Why Apps Can't Access Hardware Directly

If every app could control hardware directly, a malicious app could: turn on the camera without permission, record audio silently, read fingerprint data, disable Wi-Fi, or crash the device. Android introduces controlled layers where each request is verified before reaching hardware.

📷 Simulation: Taking a Photo (End-to-End)
1
Application Layer: User taps Capture in Camera app. The app receives the request but does NOT communicate directly with hardware.
2
Framework: Camera app calls cameraManager.openCamera(). Framework checks permissions, finds the camera service, verifies access rights.
3
ART Runtime: Java/Kotlin code executes inside ART. ART manages memory, objects, garbage collection, and application isolation.
4
Native Libraries: Framework calls optimized C/C++ libraries for camera image processing. Performance-critical operations delegated to native code.
5
HAL: Camera HAL translates generic request into hardware-specific instructions. Same API works on Samsung, Pixel, Xiaomi, OnePlus.
6
Linux Kernel: Kernel communicates with camera driver. Allocates memory, controls device I/O, ensures only authorized requests reach hardware.
7
Hardware: Camera sensor captures the image. Image travels back through the same layers to the application.

🔧 Section 3.2

Architecture Layers

Each layer of the Android architecture has specific responsibilities. Understanding each layer is essential for both development and security research.

3.2.1 — Linux Kernel

The Linux Kernel is the foundation of Android. Without it, Android cannot boot, applications cannot run, hardware cannot be accessed, and security cannot be enforced.

🐧

Kernel Responsibilities

  • Process Management — Creates processes, assigns PIDs, schedules CPU time
  • Memory Management — Allocation, release, virtual memory, process isolation
  • Device Drivers — Camera, Display, Touchscreen, Bluetooth, Wi-Fi, USB
  • File System — Manages storage operations for apps
  • Security — UIDs, process isolation, Linux permissions, SELinux
  • Power Management — CPU frequency, sleep mode, battery optimization
  • Networking — Internet access through Linux networking stack
  • IPC — Binder communication between processes
🔒

Security Perspective

  • Kernel has unrestricted access to CPU, memory, hardware, storage
  • If compromised, attacker gains control of entire device
  • Kernel vulnerabilities can lead to: privilege escalation, root access, sandbox escape
  • Key pentester questions: Kernel version? Known CVEs? Vendor drivers outdated? SELinux enforcing?
📱 Simulation: Opening Google Maps (Kernel Flow)
1
User Opens Maps: Application starts. Framework and ART prepare the process.
2
Maps Application: Requests location data. Calls Android Location Manager.
3
Framework → ART → Native → HAL: Request flows down through each layer.
4
Linux Kernel: Communicates with GPS driver. Allocates memory, controls device I/O.
5
GPS Hardware: GPS chip provides location data. Response travels back up through all layers.
⚡ Pentester's Perspective — Linux Kernel
  • Which kernel version? Check /proc/version or uname -r
  • Known CVEs? Search vulnerability databases for unpatched exploits
  • Vendor drivers? Qualcomm, MediaTek, Samsung drivers are common targets
  • SELinux enforcing? Check getenforce
  • Debug interfaces? Check for ADB root, JDWP debugging

3.2.2 — Hardware Abstraction Layer (HAL)

HAL acts as a translator between Android Framework and hardware-specific drivers. It enables the same Camera app to work on Samsung, Pixel, Xiaomi, and OnePlus — each with different camera sensors.

🔌

Why HAL Exists

  • Different phones have different camera sensors, fingerprint chips, Wi-Fi chips
  • Without HAL, the Framework would need separate code for every manufacturer
  • HAL hides hardware differences from Android applications
  • Applications never need to know which hardware is installed
🔧

HAL Modules

Audio HALSpeaker & Microphone
Bluetooth HALBluetooth Chip
Wi-Fi HALWireless Adapter
GPS HALLocation Hardware

Camera Application
↓

↓

↓

🔧

Project Treble

  • Introduced in Android 8.0
  • Standardized interface between Framework and HAL
  • Uses HIDL (Hardware Interface Definition Language)
  • Newer HALs migrating to AIDL
  • Faster Android updates, easier maintenance
🔒

Security Concerns

  • HAL is highly privileged — has access to hardware
  • Camera HAL, Fingerprint HAL, Bluetooth HAL = high-value targets
  • Vulnerable HAL can lead to privilege escalation
  • HAL vulnerabilities often affect multiple devices on same hardware platform
⚡ Pentester's Perspective — HAL
  • Is HAL outdated? Check vendor security patch levels
  • Known vulnerabilities? Camera, Bluetooth, Wi-Fi HALs historically have issues
  • Input validation? Can malformed input crash privileged services?
  • IPC abuse? Can HAL interfaces be called from untrusted apps?

3.2.3 — Native Libraries

Native Libraries are precompiled C/C++ components providing high-performance functionality. Tasks like video decoding, image rendering, cryptography, and database management require native code for performance.

📦

Key Native Libraries

  • Bionic libc — Android's C library (not glibc). Smaller, faster, mobile-optimized
  • SQLite — Local database storage. WhatsApp, Telegram, Banking apps use it
  • BoringSSL — TLS/SSL, encryption, certificate validation
  • Skia — Graphics rendering library for UI elements
  • Media Framework — Audio/video playback, recording, encoding
  • SurfaceFlinger — Display compositor combining app windows
  • OpenGL ES / Vulkan — GPU graphics rendering for games
🔒

Security Risks

  • C/C++ allow manual memory management
  • Common vulnerabilities: Buffer Overflow, Heap Overflow, Use-After-Free, Double Free
  • Native code executes close to OS — attractive attack target
  • Reverse engineering: Ghidra, IDA Pro, Radare2 for .so analysis
JNI Bridge: Java ↔ Native
Java/Kotlin Code
↓
JNI (Java Native Interface)
↓
Native Library (.so)
↓
Linux Kernel
🎯 APK Structure

Native libraries are stored in lib/ inside APKs: lib/arm64-v8a/libnative.so. The .so files are shared object libraries equivalent to Linux shared libraries.

3.2.4 — Android Runtime (ART)

ART executes Android application code. The CPU cannot execute Java/Kotlin code directly — ART converts application bytecode into machine instructions the processor understands.

🔄

ART Responsibilities

  • Code Execution — Runs application bytecode
  • Memory Management — Allocates and frees memory
  • Garbage Collection — Removes unused objects
  • Thread Management — Executes multiple tasks efficiently
  • Optimization — JIT, AOT, and Profile-Guided Optimization
  • Verification — Bytecode and class validation
🔒

Security Features

  • Application isolation via separate processes
  • Managed memory prevents many corruption bugs
  • Runtime verification of bytecode
  • Class loading validation
  • Frida, Xposed, LSPosed interact directly with ART
🔄 Simulation: APK Execution Flow
1
User Opens App: APK is loaded. classes.dex is extracted.
2
DEX Bytecode: ART reads the DEX file containing compiled application code.
3
ART Processing: AOT compiles frequently used code. JIT optimizes during execution. PGO learns usage patterns.
4
Machine Code: ART produces native machine instructions for the CPU.
5
CPU Executes: Linux Kernel schedules the process. CPU executes the compiled instructions.
6
Application Runs: UI rendered, buttons respond, app is fully functional.
🔄 Section 3.3

Runtime Deep Dive

Understanding the evolution from JVM to Dalvik to ART, and how DEX files, garbage collection, and runtime security work together.

3.3.1 — What is a Runtime Environment?

A Runtime Environment is a software layer that provides everything required for a program to execute. CPUs cannot execute Java/Kotlin directly — they only understand machine code. The Runtime bridges this gap.

Program Execution Flow
Source Code
↓
Compiler
↓
Intermediate Code
↓
Runtime Environment
↓
Machine Code
↓
CPU
Compiler (Before Execution)
  • Converts source code to intermediate format
  • Works at build time
  • Produces .class or .dex files
Runtime (During Execution)
  • Loads application classes
  • Executes code
  • Manages memory and threads
  • Performs garbage collection
  • Optimizes performance

3.3.2 — Java Virtual Machine (JVM)

The JVM is a software-based execution environment for Java programs. It enables "Write Once, Run Anywhere" by translating bytecode into platform-specific machine code.

Java Compilation and Execution
Java Source Code (.java)
↓ javac compiler
Bytecode (.class)
↓
JVM (Class Loader → Bytecode Verifier → Execution Engine)
↓
Machine Code
↓
CPU
📜

Class Loader

Loads .class files into memory. Resolves dependencies. Prepares classes for execution.

🔒

Bytecode Verifier

Validates bytecode before execution. Checks for invalid instructions, illegal memory access, corrupted bytecode.

⚙️

Execution Engine

Converts bytecode into machine instructions. Uses Interpretation and JIT compilation.

⚠️ Why Android Couldn't Use JVM

Early smartphones had 128-256MB RAM, slow CPUs, limited battery. JVM was designed for desktops/servers with higher memory usage, larger runtime, slower startup. Google needed something lighter for mobile.

3.3.3 — Dalvik Virtual Machine (DVM)

Dalvik was Android's original runtime, designed specifically for mobile devices with limited resources. It executes DEX files instead of Java .class files, using a register-based architecture for better efficiency.

Android Compilation Flow (Dalvik Era)
Java Source Code (.java)
↓ javac
Java Bytecode (.class)
↓ DX/D8 Compiler
classes.dex
↓
Dalvik Virtual Machine
↓
Machine Code → CPU
JVM (Desktop Java)
  • Stack-based architecture
  • Uses .class files
  • Higher memory usage
  • Designed for powerful systems
Dalvik (Android)
  • Register-based architecture
  • Uses .dex files (merged classes)
  • Optimized for low memory
  • Designed for mobile devices
🎯 DEX File Efficiency

500 Java classes in JVM = 500 separate .class files. In Dalvik: 500 classes merged into ONE classes.dex. Reduces duplication, storage, memory, and loading time.

3.3.4 — ART — Deep Dive

ART replaced Dalvik starting with Android 5.0. It uses AOT (Ahead-of-Time), JIT (Just-in-Time), and Profile-Guided Optimization together for faster startup and better performance.

⚡

AOT Compilation

Compiles bytecode into machine code BEFORE execution. Application starts faster. Code is ready immediately when opened.

🔄

JIT Compilation

Compiles frequently used methods DURING execution. Optimizes hot paths dynamically. Avoids compiling unused code.

📈

Profile-Guided Optimization

Monitors user behavior. Records frequently used methods and classes. Prioritizes optimization for commonly used code paths.

Dalvik
  • Mostly JIT compilation
  • Slower startup
  • Higher runtime overhead
  • Default: Android 1.0-4.4
ART
  • AOT + JIT + PGO combined
  • Faster startup
  • Better optimization
  • Default: Android 5.0+

3.3.5 — DEX Files (Dalvik Executable)

DEX files contain the compiled executable bytecode of Android applications. Every APK contains at least one classes.dex file. Large apps use MultiDEX to overcome the 64K method limit.

Complete Compilation Pipeline
Developer Writes Java/Kotlin
↓
Compiler → .class Files
↓
D8/R8 Compiler → classes.dex
↓
APK Package
↓
ART Loads DEX → Machine Code
↓
CPU Executes
📦

APK Structure

  • AndroidManifest.xml — App configuration
  • classes.dex — Executable bytecode
  • resources.arsc — Compiled resources
  • res/ — Resource files
  • lib/ — Native libraries (.so)
  • META-INF/ — Signing information
🔍

Reverse Engineering Tools

  • JADX — Decompile DEX into Java-like source
  • APKTool — Decode APK and generate Smali
  • Baksmali — Convert DEX to Smali
  • dex2jar — Convert DEX to JAR
  • Ghidra — Analyze native libraries
📖 What is Smali?

Smali is the human-readable representation of DEX bytecode — Android's assembly language. When JADX cannot fully reconstruct Java source, researchers inspect Smali code to understand application logic, authentication checks, and security routines.

3.3.6 — Garbage Collection & Memory Management

Android applications use Stack and Heap memory. ART automatically manages memory for Java/Kotlin objects. Garbage Collection removes unreachable objects from the Heap.

Stack Memory
  • Stores local variables and method calls
  • Automatically cleaned when method exits
  • Very fast access
  • Small size
  • One per thread
Heap Memory
  • Stores objects and arrays
  • Uses Garbage Collection to clean
  • Larger memory area
  • Shared within process
  • Objects remain until unreferenced
🗑️ Simulation: Object Lifecycle & Garbage Collection
1
Object Created: User user = new User(); — Memory allocated on Heap. Reference stored on Stack.
2
Object Used: Application actively uses the User object. It's reachable and stays in memory.
3
Reference Removed: user = null; — No variable points to the object. It becomes unreachable.
4
GC Detects: Garbage Collector identifies unreachable objects during its sweep cycle.
5
Memory Freed: Object removed from Heap. Memory becomes available for new objects.
⚠️ Memory Leaks

A Memory Leak occurs when an application holds references to objects no longer needed. Because references still exist, GC cannot remove them. Consequences: increased RAM usage, slower apps, battery drain, crashes. Common in Activities that retain large objects after the screen is destroyed.

3.3.7 — Runtime Security & Hooking

ART performs verification and security checks before and during execution. Tools like Frida and Xposed interact with ART to observe or modify application behavior at runtime.

🔒

Runtime Security Mechanisms

  • Class Loading — Loads only needed classes (lazy loading)
  • Bytecode Verification — Validates DEX bytecode structure
  • Runtime Verification — Continuous checks during execution
  • Managed Execution — ART controls memory, threads, objects
  • Runtime Isolation — Each app runs in separate ART process
🔍

Runtime Hooking

  • Frida — Dynamic runtime instrumentation
  • Xposed/LSPosed — Runtime module framework
  • Hook methods to observe, log, modify, or replace behavior
  • Used for: bypass root detection, disable SSL pinning, inspect values
  • Modifies behavior during execution, NOT the APK file
Static vs Dynamic Analysis
Static Analysis
APK not running
Read code
Inspect DEX
Find potential issues
Dynamic Analysis
App running
Observe execution
Inspect runtime behavior
Validate real behavior
⚡ Pentester's Perspective — Runtime
  • Which methods execute? Hook with Frida to trace method calls
  • Where are security checks? Monitor authentication logic at runtime
  • Can SSL pinning be bypassed? Use Frida to disable certificate validation
  • What values are returned? Inspect runtime variables and return values
  • Anti-hooking? Applications may detect Frida/Xposed and refuse to run
🔗 Section 3.4

Binder IPC

Android applications run in separate processes with sandboxing. Binder IPC is Android's primary mechanism for secure communication between isolated processes and system services.

3.4.1 — Why IPC is Needed

Every Android app runs in its own process with unique UID, separate memory, and private filesystem. Yet apps need to communicate with system services (GPS, Camera, Audio). IPC enables this securely.

🔗

The Problem

  • Google Maps needs GPS — but can't access hardware directly
  • Spotify needs Audio — but can't control speaker directly
  • Camera App needs Camera — but can't access sensor directly
  • Every app is isolated — how do they communicate?
🔒

The Solution: IPC

  • Applications use system services via controlled IPC
  • Permission checks enforced at every step
  • Process identity verified (UID/PID)
  • Communication happens without violating sandbox
System Services (IPC Examples)
Camera App
Google Maps
Spotify
Settings
↓ All use IPC
Camera Service
Location Service
Audio Service
Bluetooth Service

3.4.2 — What is Binder?

Binder is Android's primary IPC mechanism. It allows different processes to communicate securely, efficiently, and in a controlled manner using a client-server architecture.

🔗

Why Binder Was Created

  • Traditional IPC (pipes, sockets) not designed for mobile
  • Binder provides: high performance, low memory, permission enforcement, process identity, object-oriented communication
  • Deep integration with Android Framework
  • Optimized for mobile battery and CPU constraints
📦

Where Binder is Used

  • Camera App → Camera Service
  • Google Maps → Location Service
  • Settings → Wi-Fi Service
  • Spotify → Audio Service
  • Play Store → Package Manager
  • Almost every app interaction involves Binder
Traditional IPC
  • General-purpose
  • Limited process identity
  • Basic message passing
  • Manual security
Binder
  • Designed for Android
  • Tracks caller UID/PID
  • Object-oriented communication
  • Integrated permission checks

3.4.3 — Binder Architecture

Binder follows a Client-Server architecture with six major components: Client, Proxy, Binder Driver, Stub, System Service, and Response.

Binder Architecture
Application (Client)
↓
Binder Proxy
↓
Binder Driver (Linux Kernel)
↓
Binder Stub
↓
System Service (Server)
📱

Client

The app requesting a service. Sends parameters and waits for response. Example: Camera App requesting "Take Photo".

🔗

Proxy

Exists inside client process. Makes remote service appear local. Converts method calls into Binder transactions.

📡

Binder Driver

In Linux Kernel. Transports messages between processes. Tracks caller identity. The "post office" of Android IPC.

📦

Stub

Exists inside server process. Receives Binder transactions and converts them back into method calls.

⚙️

System Service

Performs the actual work. Camera Service, Location Service, Audio Service, etc.

🔁

Response

Result travels back: Service → Stub → Driver → Proxy → Application.

3.4.4 — Communication Flow & Service Manager

The Service Manager is Android's central registry for Binder services. Applications discover services through it, like a phonebook for system services.

📡 Simulation: Complete Binder Transaction
1
Service Registration: During boot, every system service (Camera, Audio, Location) registers with the Service Manager.
2
Service Discovery: Camera App asks Service Manager for Camera Service. Service Manager returns a Binder handle.
3
Proxy Packages: Proxy converts method call into a Binder transaction. Packages parameters.
4
Driver Transfers: Binder Driver in kernel securely moves data between processes. Tracks caller UID/PID.
5
Stub Receives: Stub unpacks the request, converts back to method call, invokes Camera Service.
6
Service Executes: Camera Service validates permissions, communicates with HAL, captures image.
7
Response Returns: Result travels back: Service → Stub → Driver → Proxy → App. User sees photo.
📡

Synchronous vs Asynchronous

  • Synchronous: App sends request, waits for response (most calls)
  • Asynchronous (One-way): App sends request, continues working (no wait)
  • Thread Pool: Services use thread pools to handle multiple concurrent requests
🔒

Binder Security

  • Caller Identity: Every transaction carries UID/PID
  • Permission Enforcement: Services validate caller permissions
  • Process Isolation: Apps cannot bypass Binder under normal conditions
  • Kernel-Mediated: Driver ensures secure message passing
⚡ Pentester's Perspective — Binder
  • Which services are exposed? List all registered Binder services
  • Permission checks? Are permissions correctly enforced?
  • Input validation? Does the service validate incoming data?
  • Exported interfaces? Are there unnecessary exposed interfaces?
  • Historical vulnerabilities? Many Android CVEs involve Binder-exposed services

Architecture from a Security Perspective

Every layer of Android's architecture introduces both functionality and security responsibilities. As security researchers, we ask: "Where can Android fail?"

↓

↓

↓
Native Libraries — Buffer overflow, memory corruption

Defense in Depth

Android doesn't rely on a single protection. It combines multiple security layers:

🔒

Application Sandboxing

Each app gets unique UID, separate process, private memory, and filesystem permissions.

🛡️

SELinux + Linux Permissions

Mandatory access control + traditional Unix permissions provide layered file system security.

🔓

Verified Boot + APK Signing

Ensures system integrity and application authenticity throughout the boot and install process.

⚡ Think Like a Pentester

Instead of asking "What does this layer do?", ask:

  • What data does this layer handle?
  • What assumptions does it make?
  • What inputs does it trust?
  • What security checks should exist?
  • What happens if those checks fail?
